🎫 OAuth Setup — Customer Admin Checklist

SAML-federated pods (Entra/Okta SSO) reject username + password for REST. Oracle's supported path is an OAuth client-credentials app that Talk2Sql exchanges for a Bearer token. Send this checklist to the customer's Oracle / identity admin — it's a one-time setup.

WHAT THE CUSTOMER ADMIN DOES (ONCE)
  1. Identify the IAM identity domain associated with the Fusion pod and its token endpoint:
    https://<iam-domain>/oauth2/v1/token
  2. Register a confidential application / OAuth client in that domain; enable the Client Credentials grant.
  3. Grant the client the Fusion REST resource scope — the exact audience string, e.g.
    urn:opc:resource:fusion:<POD>:erp/ (CX/HCM have their own segments)
  4. Associate a least-privilege service identity with read-only data roles covering only the modules to be queried.
  5. Confirm outbound network access from this workstation to both the IAM domain and the Fusion host.
WHAT THEY HAND BACK (VIA A SECURE CHANNEL)
  • Token URL · Client ID · Client Secret · Scope · Fusion base URL
THEN, IN TALK2SQL
  • Enter the four values in the OAuth fields and click Connect. The secret is stored encrypted in your OS keychain — never in profiles — and is reused automatically next time (leave the field blank).
  • If the token mints but the probe fails (HTTP 403), the scope string or data roles need adjusting on the Oracle side.
⛔ If OAuth can't be approved — fallback: request a single non-federated local Fusion service account (username + password, read-only roles, Basic auth permitted for REST) and use the 🔑 Username / Password mode.

📄 BIP SQL Engine — One-Time Setup

Runs your SQL server-side through a BI Publisher data model — real SQL with joins across any table, no REST mapping limits. Requires a one-time custom report on the pod (BI Author role needed once to create it; the connecting user only needs permission to run it).

The .xdo path shown in the BIP catalog after step 4 below
CREATE THE DATA MODEL (ONCE, IN /xmlpserver)
  1. Open BI Publisher → New → Data Model.
  2. Add a Parameter: name p_sql, type String, default value SELECT 1 AS ONE FROM DUAL.
  3. Add a SQL Query dataset (name G_1, Standard SQL) with the query exactly:
    &p_sql
    (the lexical reference substitutes your whole query — same pattern the JDBC-over-BIP drivers use)
  4. Save as /Custom/Talk2Sql/T2S_SQL_DM, then create a Report on it (any minimal layout), save as /Custom/Talk2Sql/T2S_SQL — the path above gets .xdo.
  5. Grant the connecting user (or OAuth service identity) a role that can run the report (e.g. BI Consumer + catalog permission).
🛡 The read-only gate still applies — only SELECT/WITH leaves the app. Auth: works over Basic and OAuth (not the SSO cookie mode). Results carry the same verification stamp; failures surface verbatim — never invented rows.
⚠ Large results: BIP truncates very large outputs per call — add a WHERE/ROWNUM filter rather than SELECT * on huge tables.
