Talk2Sql · Solution Architecture

Connecting to SSO-Enabled Oracle Fusion

A sanctioned authentication architecture for programmatic REST access when the target Oracle Fusion pod is federated to an external identity provider (Microsoft Entra ID, Okta, etc.). Talk2Sql-specific design — token handling, security model, and customer provisioning.
Audience · Engineering (Talk2Sql) Status · Reference design Version · 1.0 App baseline · Talk2Sql v3.0.1

Talk2Sql reads Oracle Fusion data over the REST API. On pods protected by federated single sign-on, the credential the user holds (a corporate Entra/Okta identity) cannot authorize REST calls — confirmed empirically. This document defines the authentication patterns Oracle sanctions for that situation, recommends one for Talk2Sql, and specifies exactly how to wire it into the app without weakening the anti-fabrication guarantees introduced in v3.0.0.

Design principle that carries over. Whatever the auth mode, result rows have exactly one origin: a live REST fetch in the current request, tagged with provenance. Auth changes how we obtain authorization — it never changes the rule that a failed or empty call renders an error/empty state, never fabricated data.

1 What we proved (and why workarounds end here)

Before designing, we validated the constraints against a live federated pod (Alto-Shaam, SAML-federated to Microsoft Entra ID). The findings rule out the easy paths:

Attempted pathResultWhy
Basic auth with the user's federated credentialsFailsBasic auth is validated against the pod's local identity store. A federated user's password lives at the IdP (Entra), not in Fusion — there is nothing to match.
Embedded interactive SSO, then ride the session cookie for RESTHTTP 401Interactive login succeeds, but the Fusion REST tier does not accept the OAM/UI session cookie for /fscmRestApi. (Built into Talk2Sql, tested, rejected by the pod.)
Reset the Oracle password to enable BasicN/AOn a federated pod there is no Oracle-side password; the reset flow redirects to the IdP. Resetting touches the corporate identity, not Fusion.
Conclusion. Oracle's position is explicit: federated SSO users must use an OAuth access token to access the REST APIs. Programmatic access therefore requires either a non-federated local service account (Basic) or an OAuth 2.0 token minted by the pod's IAM identity domain. There is no cookie/session shortcut.

2 The sanctioned authentication patterns

PatternIdentityFit for Talk2SqlNotes
A. Local service account
Basic auth		 Service (shared) Tactical Non-federated account created directly in Fusion's identity domain. Works with Talk2Sql's existing Basic mode today. Standing password — store in OS keychain, rotate, least-privilege roles.
B. OAuth 2.0 — Client Credentials
2-legged, Bearer		 Service (the app) Recommended A confidential client registered in the IAM identity domain mints a token scoped to the Fusion REST API. No user involvement, no standing password in the resource — secret stays at the IdP/keychain. Ideal for a read-only query tool acting as itself.
C. OAuth 2.0 — Authorization Code (PKCE)
3-legged, Bearer		 The signed-in user Optional Token minted on behalf of the actual user via the IdP login. Preserves per-user attribution and row-level/data-role security. Use when "who ran this query" must be the human, not a service identity.
D. UI session cookie — Rejected Proven not to authorize REST on a federated pod (§1). Documented here so it is not re-attempted.

3 Recommended pattern & decision tree

Default: Pattern B (OAuth 2.0 Client Credentials → Bearer token). Talk2Sql is a read-only desktop query tool; a service-identity token from the customer's IAM identity domain is the cleanest, requires no standing password in the app, and is exactly what Oracle prescribes for federated pods. Fall back to Pattern A only when the customer cannot register an OAuth client; escalate to Pattern C only when per-user identity is a hard requirement.

Figure 1 — Auth pattern decision tree. Default path for a federated pod is OAuth Client Credentials.

4 Target architecture

The Bearer token must be obtained and held in the main process, never the renderer. The renderer (the query UI) only ever asks the main process to perform an authorized GET; the client secret and the live token never enter the web context. This keeps secrets out of localStorage/DOM and lets the OS keychain protect them at rest.

Figure 2 — Component & trust boundaries. The renderer never touches the secret or the token.

5 Token flow (Client Credentials)

Figure 3 — Client-credentials token + authorized REST call, ending in the existing guard. A 401 triggers a single token refresh, then an honest error.

The token request, per Oracle's IAM identity domain:

# 2-legged client credentials — executed in the MAIN process, never the renderer
curl -X POST \
  -u "<CLIENT_ID>:<CLIENT_SECRET>" \
  -H "Content-Type: application/x-www-form-urlencoded;charset=UTF-8" \
  -d "grant_type=client_credentials&scope=urn:opc:resource:fusion:<POD>:erp/" \
  "https://<IAM_DOMAIN_HOST>/oauth2/v1/token"
# → { "access_token": "<JWT>", "token_type": "Bearer", "expires_in": 3600 }

# subsequent resource call
curl -H "Authorization: Bearer <JWT>" \
  "https://<POD>.fa.<dc>.oraclecloud.com/fscmRestApi/resources/latest/channelPrograms?limit=10"
Scope is pod- and product-specific. The audience segment (urn:opc:resource:fusion:<POD>:erp/) is provided by the customer when they register the client. CX/HCM resources use their own audience segments; the customer's IAM admin supplies the exact string.

6 Wiring it into Talk2Sql

This extends the existing additive auth model. Today Talk2Sql has authMode = "basic" | "sso" and a single transport choke point, t2sAuthGet(url), that every REST GET already flows through. Bearer support is a third mode plus a main-process token manager — the renderer and the guard are unchanged.

Renderer (unchanged contract)

  • authMode gains "oauth".
  • t2sAuthGet(url) in OAuth mode calls electronAPI.authedFetch(url) over IPC — no token in the web context.
  • Result still flows into t2sProcessResponse → provenance/empty/error. no change to the guard

Main process (new: Token Manager)

  • Reads client-id / secret / token-URL / scope from the profile; secret pulled from safeStorage.
  • Fetches & caches the token with its expires_in; refreshes proactively and once on a 401.
  • authedFetch: attach Bearer, GET via net.request, return raw {status, responseText}.

Profile schema (additive)

{
  "name": "AltoShaam DEV1 (OAuth)",
  "url":  "https://<pod>.fa.us2.oraclecloud.com",
  "authMode": "oauth",
  "oauth": {
    "tokenUrl": "https://<iam-domain>/oauth2/v1/token",
    "clientId": "<client_id>",
    "scope":    "urn:opc:resource:fusion:<pod>:erp/"
    // clientSecret is NOT stored here — it lives in the OS keychain (safeStorage)
  }
}
Invariant preserved by construction. OAuth changes only the transport. t2sProcessResponse and t2sAssertRenderable are untouched, so the guard test still proves that a dropped connection, a 401, or a zero-row response yields error/empty — never fabricated rows — regardless of auth mode.

7 Security model

ControlImplementation
Secret at restClient secret encrypted via Electron safeStorage (OS keychain / DPAPI). Never in localStorage, profile JSON, or the renderer.
Secret in motionToken request executed only in the main process over TLS to the IAM domain. Renderer requests authorization by URL, not by credential.
Token handlingHeld in main-process memory; honored until expires_in; single silent refresh on 401; cleared on disconnect.
Least privilegeClient granted read-only scope; the underlying service identity carries only the data roles needed for the modules in scope.
NetworkTLS to both the IAM token endpoint and the Fusion resource host; pin allowed hosts to the configured pod + domain.
AuditabilityEvery rendered result already carries a provenance stamp (instance, timestamp, row count). With Pattern C, the token also attributes the call to the human user in Fusion's audit.
Failure postureAuth failure → honest error state. No path substitutes cached, sample, or model-generated rows (enforced by the v3.0.0 guard).

8 Customer provisioning checklist

What the customer's Oracle / identity admin must do once, so Talk2Sql connects on the first try. Hand this to their IT/security team.

  • Identify the IAM identity domain associated with the Fusion pod and its token endpoint https://<iam-domain>/oauth2/v1/token.
  • Register a confidential application / OAuth client in that domain; enable the Client Credentials grant. (Add Authorization Code + PKCE only if Pattern C / per-user is required.)
  • Grant the client the Fusion REST resource scope — provide the exact audience string, e.g. urn:opc:resource:fusion:<POD>:erp/ (and CX/HCM equivalents if needed).
  • Associate a least-privilege service identity / data roles covering only the modules to be queried (read-only).
  • Provide back: Client ID, Client Secret (via a secure channel), Token URL, Scope, and the Fusion base URL.
  • Confirm outbound network access from the workstation to both the IAM domain and the Fusion host.
If OAuth cannot be approved (Pattern A fallback): request a single non-federated local Fusion service account — username, password (secure channel), read-only data roles, and confirmation that Basic auth is permitted for REST on that account.

Bottom line. For a federated pod, register an OAuth client-credentials application in the customer's IAM domain and let Talk2Sql exchange it for a Bearer token in the main process. It is Oracle-sanctioned, stores no standing password in the app, keeps the secret in the OS keychain, and — because every call still lands in the same guard — it cannot reintroduce fabricated data. The session-cookie route is closed; this is the professional path.
